考试题解析

一、Java中有哪些类加载器？说说什么是双亲委派？哪些场景打破了双亲委派机制？

  • 有哪些类加载器

    引导类加载器：负责加载存放在JDK\jre\lib目录中的类，用C++实现，在java中打印输出为：null

    扩展类加载器：负责加载JDK\̲j̲r̲e̲\lib\ext目录中的类，java实现，在java中打印输出为：ExtClassLoader

    应用类加载器：负责加载应用程序自己的代码，java实现，在java中打印输出为：AppClassLoader

  • 什么是双亲委派

    双亲委派：当一个类加载器收到需要加载的请求时，它自己不会先去加载，而是将这个请求交给父类的加载器去加载，如果父类的加载器还有父类，那就一直向上请求直到请求到顶级的加载器

    作用：避免类的重复加载，保证程序的安全

  • 打破双亲委派机制

    tomcat容器，一个tomcat容器可以部署多个项目，可能很多类需要被重复加载

二、GC是如何判断对象可以被回收的？

  • 引用计数法：每个对象有一个引用计数属性，新增一个引用时计数加1，引用释放时计数减1，计数为0时可以回收
  • 可达性分析法：从GCRoots开始向下搜索，搜索所走过的路径称为引用链。当一个对象到GCRoots没有任何引用链相连时，则证明此对象是不可用的，那么虚拟机就判断是可回收对象

三、线程的生命周期？阻塞情况有几种？

  • 线程的生命周期

    新建状态（New）：新创建了一个线程对象

    就绪状态（Runnable）：线程对象创建后，其他线程调用了该对象的start方法。该状态的线程位于可运行线程池中，变得可运行，等待获取CPU的使用权

    运行状态（Running）：就绪状态的线程获取了CPU，执行程序代码

    阻塞状态（Blocked）：阻塞状态是线程因为某种原因放弃CPU使用权，暂时停止运行。直到线程进入就绪状态，才有机会转到运行

    死亡状态（Dead）：线程执行完了或者因异常退出了run方法，该线程结束生命周期

  • 阻塞情况

    等待阻塞：运行的线程执行wait方法，该线程会释放占用的所有资源，JVM会把该线程放入“等待池”中。进入这个状态后，是不能自动唤醒的，必须依靠其他线程调用notify或notifyAll方法才能被唤醒，wait是object类的方法

    同步阻塞：运行的线程在获取对象的同步锁时，若该同步锁被别的线程占用，则JVM会把该线程放入“锁池”中

    其他阻塞：运行的线程执行sleep或join方法，或者发出了I/O请求时，JVM会把该线程置为阻塞状态。当sleep状态超时、join等待线程终止或者超时、或者I/O处理完毕时，线程重新转入就绪状态。sleep是Thread类的方法

四、谈谈你对SpringIOC的理解？

springIOC是spring框架十分重要的特性之一，一般我们称之为控制反转，要理解IOC，就需要理解什么是控制反转？控制了什么？什么发生了反转？

  • 控制：没有容器之前，创建对象是有程序员创建，通过new关键字或者反射技术。那么有了容器之后，我们如果要创建对象，只要配置好或者添加相关的注解，spring就会帮我们创建对象，并将创建好的对象放到spring容器中进行管理，所以，spring控制了对象的创建权

  • 反转：在说反转的时候就需要提到一个很重要的特性，依赖注入。比如A类中依赖B类，那么在创建A类的时候，就会判断B类在不在容器中，如果不存在的话就会创建B类。

五、在你的工作过程中，有哪些Sql优化的经历？

SQL优化的经历需要问清楚，是SQL语句层面的优化，还是整个该SQL业务流的优化

  • 业务层面

    比如说查询一些日志，或者交易记录，只能一段时间内查询，查询三个月之间的数据

  • 代码层面

    可以将复杂的多表关联查询，拆分为多个查询语句

  • SQL层面

    主要是索引相关的优化，比如创建联合索引，避免索引失效的查询语句，尽量以数据量小的表作为驱动表，模糊查询过程中尽量不在前面添加%

六、Redis过期key的删除策略？

  • 惰性删除：只有当访问一个key时，才会判断该key是否已过期，过期则清除。该策略可以最大化地节省CPU资源，却对内存非常不友好。极端情况可能出现大量的过期key没有再次被访问，从而不会被清除，占用大量内存
  • 定期删除：每隔一定的时间，会扫描一定数量的数据库的expires字典中一定数量的key，并清除其中已过期的key。该策略是一个折中方案。通过调整定时扫描的时间间隔和每次扫描的限定耗时，可以在不同情况下使得CPU和内存资源达到最优的平衡效果

七、谈谈你对Redis事务的理解？

  • 开始事务：multi
  • 命令入队：如果执行的不是（multi,exec,discard,watch）则会将命令添加到队列中 set name zhangsan
  • 事务执行：
    • exec：执行事务
    • discard：清空事务队列，并放弃执行事务，客户端从事务状态中退出
  • 特点：redis事务不支持回滚机制，但是它会检查每一个事务中的命令是否错误
命令 描述
multi 开始事务 用于标记事务块的开始。Redis会将后续的命令逐个放入队列中，然后才能使用EXEC命令原子化地执行这个命令序列。
exec 执行事务 在一个事务中执行所有先前放入队列的命令，然后恢复正常的连接状态。 当使用WATCH命令时，只有当受监控的键没有被修改时，EXEC命令才会执行事务中的命令，这种方式利用了检查再设置（CAS）的机制。 这个命令的返回值是一个数组，其中的每个元素分别是原子化事务中的每个命令的返回值。 当使用WATCH命令时，如果事务执行中止，那么EXEC命令就会返回一个Null值。
discard 回滚 清除所有先前在一个事务中放入队列的命令，然后恢复正常的连接状态。 如果使用了WATCH命令，那么DISCARD命令就会将当前连接监控的所有键取消监控。
watch 监控指定的key 当某个事务需要按条件执行时，就要使用这个命令将给定的键设置为受监控的。
unwatch 撤销监控

八、Nginx有哪些作用？负载均衡有哪些算法？描述一下每个算法的优劣势？

  • 作用：web服务器，反向代理，负载均衡，动静分离
  • 负载均衡：
    • 轮询法:每个请求按时间顺序逐一分配到不同的后端服务，如果后端某台服务器死机，自动剔除故障系统，使用户访问不受影响
    • 加权轮询:weight的值越大分配到的访问概率越高，主要用于后端每台服务器性能不均衡的情况下
    • ip_hash:每个请求按访问IP的哈希结果分配，使来自同一个IP的访客固定访问一台后端服务器，并且可以有效解决动态网页存在的session共享问题
    • fair:fair算法可以根据页面大小和加载时间长短智能地进行负载均衡，也就是根据后端服务器的响应时间来分配请求，响应时间短的优先分配。Nginx本身不支持fair，如果需要这种调度算法，则必须安装upstream_fair模块
    • url_hash:按访问的URL的哈希结果来分配请求，使每个URL定向到一台后端服务器，可以进一步提高后端缓存服务器的效率。Nginx本身不支持url_hash，如果需要这种调度算法，则必须安装Nginx的hash软件包

九、你在工作中设计接口的如何保证幂等性？

  • 唯一id。每次操作，都根据操作和内容生成唯一的id，在执行之前先判断id是否存在，如果不存在则执行后续操作，并且保存到数据库或者redis等
  • 服务端提供发送token的接口，业务调用接口前先获取token,然后调用业务接口请求时，把token携带过去,务器判断token是否存在redis中，存在表示第一次请求，可以继续执行业务，执行业务完成后，最后需要把redis中的token删除
  • 建去重表。将业务中有唯一标识的字段保存到去重表，如果表中存在，则表示已经处理过了
  • 版本控制。增加版本号，当版本号符合时，才能更新数据
  • 状态控制。例如订单有状态已支付 未支付 支付中 支付失败，当处于未支付的时候才允许修改为支付中等

十、什么是熔断？什么是服务降级？

  • 服务熔断：服务熔断的作用类似于我们家用的保险丝，当某服务出现不可用或响应超时的情况时，为了防止整个系统出现雪崩，暂时停止对该服务的调用
  • 服务降级：是从整个系统的负荷情况出发和考虑的，对某些负荷会比较高的情况，为了预防某些功能（业务场景）出现负荷过载或者响应慢的情况，在其内部暂时舍弃对一些非核心的接口和数据的请求，而直接返回一个提前准备好的fallback（退路）错误处理息。这样，虽然提供的是一个有损的服务，但却保证了整个系统的稳定性和可用性

    • url_hash:按访问的URL的哈希结果来分配请求，使每个URL定向到一台后端服务器，可以进一步提高后端缓存服务器的效率。Nginx本身不支持url_hash，如果需要这种调度算法，则必须安装Nginx的hash软件包

九、你在工作中设计接口的如何保证幂等性？

  • 唯一id。每次操作，都根据操作和内容生成唯一的id，在执行之前先判断id是否存在，如果不存在则执行后续操作，并且保存到数据库或者redis等
  • 服务端提供发送token的接口，业务调用接口前先获取token,然后调用业务接口请求时，把token携带过去,务器判断token是否存在redis中，存在表示第一次请求，可以继续执行业务，执行业务完成后，最后需要把redis中的token删除
  • 建去重表。将业务中有唯一标识的字段保存到去重表，如果表中存在，则表示已经处理过了
  • 版本控制。增加版本号，当版本号符合时，才能更新数据
  • 状态控制。例如订单有状态已支付 未支付 支付中 支付失败，当处于未支付的时候才允许修改为支付中等

十、什么是熔断？什么是服务降级？

  • 服务熔断：服务熔断的作用类似于我们家用的保险丝，当某服务出现不可用或响应超时的情况时，为了防止整个系统出现雪崩，暂时停止对该服务的调用
  • 服务降级：是从整个系统的负荷情况出发和考虑的，对某些负荷会比较高的情况，为了预防某些功能（业务场景）出现负荷过载或者响应慢的情况，在其内部暂时舍弃对一些非核心的接口和数据的请求，而直接返回一个提前准备好的fallback（退路）错误处理息。这样，虽然提供的是一个有损的服务，但却保证了整个系统的稳定性和可用性

    • 使用多路 I/O 复用模型，充分利用CPU资源

四、如何保证Redis缓存中与数据库中的数据一致性？

1）先删除缓存，在写数据库

存在的问题：

在高并发场景下，当第一个线程删除了缓存，还没有来得及写数据库，第二个线程来读取数据，发现缓存中没有数据，则会去数据库中读取数据（此时读到的数据是旧的数据，也就是脏数据）。第二个线程将读到的数据写入到缓存中（如果此时第一个线程已经将修改后的数据写到缓存中了），那么缓存中保存的还是旧的数据，也就是脏数据。存在数据不一致问题

解决方案：

1. 对数据进行修改后，不删除缓存，而是将缓存中的数据设置为一个默认值（如：-9999），当线程来读取数据时，发现返回值为默认值，则对该线程进行休眠一会儿（等待第一个线程将修改数据库之后的值，设置到缓存中），然后再重新查询缓存。对业务会有一定的入侵。
2. 延时双删。先删除缓存，在修改数据库。该线程休眠一会儿，在重新删除缓存（保存数据的一致性，存在一定时间的脏数据）
2）先写数据库，在删缓存

存在的问题：

当删除缓存失败时，会存在数据不一致问题。

解决方案：

 1. 给缓存设置一个有效期
 2. 引入MQ，添加事务，保证写数据库与删除缓存的原子操作（对于这种情况，一种常见的解决方案就是使用消息中间件来实现删除的重试。MQ一般都自带消费失败重试的机制，当要删除缓存的时候，就往 MQ中扔一条消息，缓存服务读取该消息并尝试删除缓存，删除失败了就会自动重试）。

总结：不管哪种方式，最终目的是要保证数据的最终一致性，就是缓存中的数据与数据库中的数据一致。但是不可避免存在一定时间内的脏数据问题。

所以，引用缓存的数据，修改不能太频繁。

六、@Autowired注解与@Resource注解的区别？

  • @Autowired
    • spring框架提供的注解，需要在spring体系中使用
    • @Autowired默认按照类型注入，如果有该类型在容器中有多个，则按照名称注入（按照属性的名称来注入），如果没有找到该属性的名称对应的bean实例，则报错

    • JDK提供的注解，不依赖于spring体系，允许注入null

    • 指定name：从Spring上下文中查找名称（id）匹配的bean进行装配，找不到则抛出异常
    • 指定type：从Spring上下文中找到类似匹配的唯一bean进行装配，找不到或是找到多个，都会抛出异常
    • 同时指定name和type：从Spring上下文中找到唯一匹配的bean进行装配（与指定name一样），找不到则抛出异常

十、请描述一下SpringMVC处理请求的流程？

  1. 浏览器发送请求至应用程序，被前端控制器（DispatcherServlet）接收；
  2. 前端控制器调用处理器映射器（HandlerMapping）查找处理器（Handler）；
  3. 处理器映射器根据请求映射从Spring容器中获取得到处理器（Handler，也是自定义的Controller）；
  4. 处理器映射器组装处理器执行链（HandlerExecutionChain），返回给前端控制器；
  5. 前端控制器找到合适的处理器适配器（HandlerAdapter）；
  6. 处理器适配器执行处理器，处理器（Controller）执行完成后返回模型和视图（ModelAndView）；
  7. 处理器适配器把模型和试图返回给前端控制器；
  8. 前端控制器调用视图解析器（ViewReslover）解析模型和视图（ModelAndView）；
  9. 视图解析器解析完成后，返回具体的视图（View）；
  10. 前端控制器进行视图渲染（即把模型数据填充到视图中），最终响应浏览器的请求；

  • 有状态就是有数据存储功能

九、AOP的底层实现是什么？有几种方式？这几种方式有什么区别？

  • AOP的底层实现是动态代理
  • 基于JDK的动态代理，基于CGLIB的动态代理
  1. 原理区别
    • JDK动态代理是利用反射机制生成一个实现代理接口的匿名类，在调用具体的方法之前调用InvocationHandler
    • CGLIB动态代理是利用ASM开源包，对被代理类的class文件进行字节码修改，生成被代理类的子类，覆盖目标方法。
  2. 依赖区别
    • JDK动态代理，不需要第三方的库，只要JDK环境就可以；
    • CGLIB动态代理，需要加入cglib、asm的类库
  3. 要求区别
    • JDK动态代理，被代理的类必须有实现的接口
    • CGLIB动态代理，针对具体的实现类进行代理，但是类不能用final修饰。
  4. 编程区别
    • JDK动态代理，需要实现InvocationHandler接口编写增强、使用Proxy.newInstance获取代理对象
    • CGLIB动态代理，实现MethodInterceptor接口中的invoke方法实现增强，使用Enhancer根据被代理对象和增强修改字节码文件

十、Beanfactory和Factorybean的区别？

  • Factorybean是一个bean对象，该对象主要用于定制一些复杂bean对象的实例化，比如创建代理对象

二、请描述一下spring中默认的事务传播行为？

传播行为 描述
PROPAGATION_REQUIRED 如果当前存在事务，则加入该事务；
如果当前没有事务，则创建一个新的事务。【默认】
PROPAGATION_SUPPORTS 如果当前存在事务，则加入该事务；
如果当前没有事务，则以非事务的方式继续运行。
PROPAGATION_MANDATORY 如果当前存在事务，则加入该事务；
如果当前没有事务，则抛出异常。
PROPAGATION_REQUIRES_NEW 创建一个新的事务，如果当前存在事务，则把当前事务挂起。
PROPAGATION_NOT_SUPPORTED 以非事务方式运行，如果当前存在事务，则把当前事务挂起。
PROPAGATION_NEVER 以非事务方式运行，如果当前存在事务，则抛出异常。
PROPAGATION_NESTED 如果当前存在事务，则在嵌套事务内执行。
如果当前没有事务，则执行 REQUIRED 类似的操作

十、请描述一下mybatis的缓存机制？不同的缓存有什么特点？

Mybatis中的缓存是为了减少对数据库的访问，提交资源的利用率和响应效率。Mybatis支持的缓存有两个级别，SqlSession内级别（一级缓存）和SqlSession间（Mapper）级别（二级缓存）

缓存级别 是否默认开启 描述
一级缓存 默认开启 在同一个SqlSession中进行多次同样的查询操作，只有第一次从数据库中查询数据，而后的其他几次会直接从一级缓存中读取数据
二级缓存 需手动开启 在不同的SqlSession中进行同样的查询操作，只有最先执行的一次查询操作会从数据库中查询，其他几次会直接从二级缓存中读取数据
