锁级别

synchronized会根据线程数量决定加锁的方式(等级)

有4种锁等级: 无锁(无线程) 偏向锁(1线程) 轻量级锁(双线程) 重量级锁(三个线程以上)

下面的线程必须被synchronized修饰

无锁: 对象刚创建没有被线程使用时处于无锁状态(偏向位0, 标志01)

偏向锁: 对象被一个线程操作时, 处于偏向锁状态(偏向位1, 标志01)

轻量锁: 两个线程需要操作同一对象时, 升级于轻量锁状态, 偏向位空,标志00 并且使用CAS算法进行两个线程间的锁交换

重量锁: 三个以上的线程都需要操作同一对象时升级为重量锁状态, 偏向位为空, 标志10, 并且不再使用CAS算法而是使用底层CPU的进程同步算法进行锁交换

同步锁只能升级, 一旦升级不会再降级

线程安全

线程安全的本质是对资源访问是否具有独占性问题

多个线程同时对资源进行写入和读取操作时会发生线程不安全

如何保证线程安全:

确保资源的独占性: 同一时刻只能有一个线程在访问临界资源

也被称为序列化访问临界资源方案或者称为同步互斥访问

java中主要有两种线程锁机制来实现同步互斥访问的方法: synchronized Lock

Synchronized(互斥锁)

synchronized被称为互斥锁, 是Java中的一个关键字

synchronized可以用于标记方法或代码块

synchronized特性:

  1. 互斥: 对象锁只能被一个线程获得, 不能被其他线程抢占
  2. 重入: 一个线程获得对象锁后, 该线程的不同方法可以抢占这个对象锁

java中每个对象都有monitor属性, 也被称为锁标记或监视器

当多线程执行体试图访问某个对象时, 需要先获取该对象的monitor才能访问

当synchronized标记的代码块或者方法执行时需要先获取指定对象的monitor, 其他需要获取monitor的线程会被锁定直到monitor被释放才能继续执行

只有整个代码块或方法结束才会释放锁定的monitor

通常使用synchronized代码块, 更加灵活

同步锁的代码量越少, 多线程效率越高

Synchronized的使用方法

锁升级是面试时需要掌握的知识

synchronized会根据线程数量决定加锁的方式(等级)

有4种锁等级: 无锁(无线程) 偏向锁(单线程) 轻量级锁(多个线程) 重量级锁(自旋四次以上)

锁等级是在synchronized中实现,与Lock接口无关

无锁: 对象刚创建没有被同步代码块使用时处于无锁状态(偏向位0, 标志位01)

偏向锁: 有一个线程访问同步块时, 升级为偏向锁状态(偏向位1, 标志01)

轻量锁: 两个以上线程需要操作同一对象时存在锁竞争, 升级于轻量锁状态, 偏向位空,标志00 并且使用CAS算法进行两个线程间的锁交换(锁自旋)

重量锁: 轻量锁处于锁竞争状态, 如果通过锁自旋机制失败十次(默认)会升级为重量锁状态, 偏向位为空, 标志10, 并且不再使用CAS算法而是使用底层CPU的进程同步算法进行锁交换

锁自旋是一种锁竞争机制, 使用CAS算法(乐观锁)进行锁交换, 当交换失败十次以上时会导致轻量级锁升级为重量级锁

总结

  1. synchronized支持同步块和同步方法
  2. 不同monitor对象的区别需要重点掌握
  3. Lock框架采用CAS原理, 提供公平选择和非公平选择取锁
  4. ThreadLocal负责管理资源副本, 不直接控制线程, 会为每个线程都产生一个独立的关联资源对象

ReentrantLock特性:

  1. 可重入锁
  2. 采用递归无阻塞同步机制(乐观锁)
  3. 乐观锁实现, 与synchronized的行为类似, 但是比syn更灵活更强大, 增加了轮询 超时 中断等高级功能, 支持设置公平锁和非公平锁(默认)

重入锁创建步骤

  1. 创建重入锁对象
  2. 创建Runnable对象并执行, 将重入锁对象传给Runnable对象
  3. 执行lock方法或trylock获取锁
  4. 任务执行完后执行unlock()释放锁

公平锁和非公平锁

公平锁: 所有线程放入队列中排队获取锁, 保证顺序

优点: 所有线程都能获取到资源

缺点: 吞吐量低, 同一时刻只有一个线程允许

Lock接口对比synchronized的优势

  1. 可以跨方法实现锁, 比如在方法1中调用lock, 然后方法2中调用unlock()
  2. 可以用trylock(超时时间) 尝试获取资源锁
  3. 支持公平锁

  2. synchronized在产生异常时也能正常释放锁

Thread.sleep和Object.wait()的区别:

  1. sleep会暂停当前线程的执行, sleep期间该不能被手动唤醒, 并且不会释放monitor资源
  2. wait()也会暂停当前线程的执行, 可以被notify手动唤醒, 并且会将占用的monitor资源释放

相似之处:

  1. 都会线程进入Not Runnable状态
  2. 都是底层由C语言写的native方法

sleep状态的线程如果被中断会引发InterruptedException异常

java中还有一个线程安全关键字volatile

volatile用于修饰变量, 作用有:

  1. 保证内存可见性
  2. 防止指令重排序

当一个变量被volatile修饰时, 线程对变量的操作将写入缓存, 并同步到主存中, 然后基于CPU缓存一致性(MESI)协议来更新其他线程的变量值

volatile不能严格保证线程安全, 只能减少一部分的同步问题

  1. 任务队列: 使进程按照FIFO(先进先出)的队列模式进行排队

  3. 任务对接接口: 线程池接收任务的接口
  4. 线程管理器: 管理线程池中每个线程的生命周期

FIFO是队列结构的特性

线程池在不同任务量下的状态:

  1. 线程池初始化指定数量的线程
  2. 线程池中有空闲线程时, 直接分配任务给线程处理
  3. 线程都在处理任务的状态下, 将后续的任务进入任务队列
  4. 当线程都在工作状态, 并且任务队列也满了时, 扩增线程池的线程数量
  5. 当线程池的线程数量已经达到最大值, 当线程都在工作状态, 并且任务队列也满了时,开启任务拒绝策略

死锁

死锁是指多线程运作时发生了锁无法被自动解开, 无限期锁死的情况

死锁发生的条件:

  1. 一定是互斥锁被使用
  2. 锁的占用是无限期, 没有时效
  3. 锁无法被其他线程抢占, 只能被一个线程主动释放
  4. 多个线程保持死锁状态

死锁发生的根本原因是业务逻辑有问题

如何解决死锁:

  1. 选用时效锁,如redis
  2. 统一加锁顺序

Daemon thread(守护线程)

java中的daemon thread(守护线程)是运行在后台, 负责管理或者辅助其他线程运行的线程

java中的GC(垃圾回收器)也是一个守护线程

特点:

  1. 守护线程是极低优先级线程, 当所有用户线程都运行结束时, JVM会自动关闭而不会考虑守护线程的运行状态
  2. JVM会强制关闭守护进程, 不会考虑其运行状态

当JVM中最后一个非守护进程运行结束时, 所有守护进程会被强制结束

Thread类中提供了实例方法setDaemon(boolean flag)用于将该线程对象设为守护进程或非守护进程

多线程实现

java中实现多线程一般有两种方式: 继承Thread类或者实现Runnable接口

  1. 继承Thread类, 并重写该类的run方法, 通过调用Thread类的实例方法start()启动线程, 线程启动时会执行重写后的run方法
  2. 实现Runnable接口, 并重写run方法, 也是调用实例方法start()启动

Thread和Runnable的区别:

  1. 继承Thread后无法再继承其他类, 实现Runnable接口后还能继承其他类
  2. Thread类还提供了yield() interrupt()等方法实现线程的基本功能, Runnable接口没有提供(接口不能有具体方法实现)
  3. runnable实现类对象可以在多个线程间共享

  4. Waiting

  6. Terminated

死锁出现在并发编程中, 当多个线程都因为等待其他线程释放资源而无法继续运行时会导致死锁产生

死锁的四个必要条件

死锁产生需要有四个必要条件, 缺一不可

  1. 互斥: 一个资源只能被一个线程占用直到被该线程释放
  2. 资源保留: 当线程请求新资源时, 不需要释放已经占用的资源
  3. 无强占: 线程占用的资源只能被自身释放, 不能被其他线程强占
  4. 循环等待: 所有线程都在等待另一个线程占用的资源, 同时占用资源的线程也在等待其他线程占用的资源被释放

如何避免死锁

死锁处理主要有三种方法:

  1. 探测
  2. 预防
  3. 避免

探测: 当探测到线程死锁产生时强制结束进程

一般被操作系统使用, 会导致线程异常结束

预防死锁只需要将产生死锁的四个必要条件之一破坏掉即可

  1. 互斥: 破坏互斥只能将锁取消掉, 因为这是保证线程安全的条件, 也是线程锁设计的目的 (临界资源互斥) , 编程中较少使用
  2. 资源保留: 将所有需要占用的资源一次性全部申请, 会导致线程并发能力降低, 效率变低, 编程中较少使用
  3. 资源强占: 锁可以被其他线程强占, 会导致线程执行出问题, 较少使用
  4. 循环等待: 使线程按照相同的顺序请求资源, 可以破坏死锁

代码实现, 修改线程B的代码

避免死锁: 对代码进行分析, 及时释放不需要的资源, 尽量避免产生死锁的情况

设置资源占用超时时间, 超时强制释放资源结束线程

==和equals的区别

  1. ==是操作符一般用于基本数据类型的对比，如果两个操作数是对象时会比较两个对象的内存地址是否相同
  2. equals是方法，Object类的equals和==一样是对比两个对象的内存地址
  3. 包装类重写了Object类的equals方法，对值进行对比
  4. 用equals时需要同时对hashcode方法进行重写

String StringBuffer StringBuilder的区别

  1. String StringBuilder StringBuffer都是java中对字符串进行封装的类
  2. String对象一旦创建不可改变，每次字符串拼接都会创建新的String对象，String类对象有缓存池管理，只有用new创建新的String对象才能不通过缓存池
  3. StringBuilder和StringBuffer和类类似，创建的对象都是可变的，区别在于StringBuffer是线程安全的，StringBuilkder线程不安全，性能较高
